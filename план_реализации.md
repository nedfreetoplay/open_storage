\# 🗂️ План программы для хранения неизменяемых файлов (Hydrus-подобная архитектура)



> \*\*Цель\*\*: Система хранения файлов с идентификацией по хэшу (SHA256/MD5), системой тегов и клиент-серверной архитектурой на Python 3.14 + SQLite, с возможностью миграции на Rust.



\---



\## 🔧 Архитектурная схема



```

┌─────────────────┐     ┌─────────────────────┐     ┌─────────────────┐

│   GUI Client    │────▶│   Server Core       │────▶│   SQLite DB     │

│   (PyQt/PySide) │ IPC │   (FastAPI/Custom)  │     │   + File Store  │

└─────────────────┘     └─────────────────────┘     └─────────────────┘

&#x20;        │                       │

&#x20;        │    ┌──────────────────┘

&#x20;        ▼    ▼

┌─────────────────┐

│ Remote Client   │

│ (HTTP API)      │

└─────────────────┘

```



\### Режимы работы:

| Режим | Описание | Транспорт |

|-------|----------|-----------|

| \*\*Local\*\* | Клиент запускает сервер как subprocess, общение через Unix socket / named pipe | IPC (ZeroMQ/asyncio streams) |

| \*\*Remote\*\* | Сервер запущен отдельно, клиенты подключаются по сети | HTTP/REST + WebSocket |

\## 🗄️ Схема базы данных (SQLite)



\### Основные таблицы:



```sql

\-- 1. Файлы: хранение хэшей и метаданных

CREATE TABLE files (

&#x20;   hash\_id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   sha256 BLOB NOT NULL UNIQUE,        -- 32 байта, плотное хранение

&#x20;   md5 BLOB,                           -- опционально, 16 байт

&#x20;   file\_size INTEGER NOT NULL,

&#x20;   mime\_type TEXT,

&#x20;   width INTEGER, height INTEGER,      -- для изображений/видео

&#x20;   duration\_ms INTEGER,                -- для медиа

&#x20;   import\_timestamp REAL NOT NULL,     -- UNIX timestamp

&#x20;   file\_path TEXT,                     -- относительный путь в хранилище

&#x20;   status TEXT CHECK(status IN ('active', 'deleted', 'quarantined')) DEFAULT 'active'

);



\-- 2. Теги: иерархическая система с неймспейсами (как в Hydrus)

CREATE TABLE namespaces (

&#x20;   namespace\_id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   namespace TEXT UNIQUE NOT NULL      -- 'person', 'artist', 'copyright', ''

);



CREATE TABLE subtags (

&#x20;   subtag\_id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   subtag TEXT NOT NULL                -- 'andreas', 'pixiv', 'original'

);



CREATE TABLE tags (

&#x20;   tag\_id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   namespace\_id INTEGER NOT NULL REFERENCES namespaces(namespace\_id),

&#x20;   subtag\_id INTEGER NOT NULL REFERENCES subtags(subtag\_id),

&#x20;   UNIQUE(namespace\_id, subtag\_id)     -- один тег = одна пара

);



\-- 3. Связь файлов и тегов (junction table)

CREATE TABLE file\_tag\_mappings (

&#x20;   mapping\_id INTEGER PRIMARY KEY AUTOINCREMENT,

&#x20;   hash\_id INTEGER NOT NULL REFERENCES files(hash\_id) ON DELETE CASCADE,

&#x20;   tag\_id INTEGER NOT NULL REFERENCES tags(tag\_id) ON DELETE CASCADE,

&#x20;   confidence REAL DEFAULT 1.0,        -- 0.0-1.0 для авто-тегов

&#x20;   source TEXT,                        -- 'manual', 'downloader', 'ai'

&#x20;   added\_timestamp REAL NOT NULL,

&#x20;   UNIQUE(hash\_id, tag\_id)             -- один тег на файл один раз

);



\-- 4. Индексы для производительности поиска

CREATE INDEX idx\_files\_sha256 ON files(sha256);

CREATE INDEX idx\_files\_status ON files(status);

CREATE INDEX idx\_mappings\_hash ON file\_tag\_mappings(hash\_id);

CREATE INDEX idx\_mappings\_tag ON file\_tag\_mappings(tag\_id);

CREATE INDEX idx\_tags\_namespace ON tags(namespace\_id);

CREATE INDEX idx\_subtags\_text ON subtags(subtag);

```



\### Дополнительные таблицы (опционально, по мере развития):



```sql

\-- Алиасы и синонимы тегов (для поиска)

CREATE TABLE tag\_siblings (

&#x20;   sibling\_id INTEGER PRIMARY KEY,

&#x20;   tag\_id INTEGER NOT NULL REFERENCES tags(tag\_id),

&#x20;   master\_tag\_id INTEGER NOT NULL REFERENCES tags(tag\_id),

&#x20;   UNIQUE(tag\_id)

);



\-- Иерархия тегов (родитель-потомок)

CREATE TABLE tag\_parents (

&#x20;   parent\_id INTEGER PRIMARY KEY,

&#x20;   tag\_id INTEGER NOT NULL REFERENCES tags(tag\_id),

&#x20;   parent\_tag\_id INTEGER NOT NULL REFERENCES tags(tag\_id),

&#x20;   UNIQUE(tag\_id, parent\_tag\_id)

);



\-- Кэш поисковых запросов

CREATE TABLE search\_cache (

&#x20;   cache\_id INTEGER PRIMARY KEY,

&#x20;   query\_hash BLOB NOT NULL UNIQUE,    -- хэш параметров поиска

&#x20;   result\_hash\_ids BLOB,               -- сжатый список hash\_id

&#x20;   created\_at REAL NOT NULL,

&#x20;   expires\_at REAL

);

```



> 💡 \*\*Важно\*\*: Хэши хранятся как `BLOB` (32 байта для SHA256), а не как HEX-строки — это экономит 50% места и ускоряет сравнения \[\[11]].

\## 🔌 Интерфейсы и протоколы



\### 1. Unified API Client (абстракция транспорта)



```python

\# shared/protocols.py

from abc import ABC, abstractmethod

from typing import AsyncIterator, Optional

from .types import HashID, TagID, FileMetadata, TagMapping



class StorageBackend(ABC):

&#x20;   @abstractmethod

&#x20;   async def add\_file(self, file\_path: Path, tags: list\[str]) -> HashID: ...

&#x20;   @abstractmethod

&#x20;   async def get\_file(self, hash\_id: HashID) -> Optional\[FileMetadata]: ...

&#x20;   @abstractmethod

&#x20;   async def search(self, tags: list\[str], exclude: list\[str] = None) -> list\[HashID]: ...

&#x20;   @abstractmethod

&#x20;   async def add\_tag(self, hash\_id: HashID, tag: str) -> bool: ...

&#x20;   @abstractmethod

&#x20;   async def remove\_tag(self, hash\_id: HashID, tag: str) -> bool: ...

```



\### 2. Реализации:



| Транспорт | Класс | Особенности |

|-----------|-------|-------------|

| \*\*Local IPC\*\* | `IPCClient` | AsyncIO Unix socket / Windows named pipe, zero-copy для больших файлов |

| \*\*HTTP\*\* | `HTTPClient` | FastAPI + httpx, streaming для загрузки/скачивания, JWT auth |



\### 3. Пример REST API (сервер):



```http

POST /api/v1/files          # Загрузка файла → {hash\_id, sha256}

GET  /api/v1/files/{hash\_id} # Метаданные файла

POST /api/v1/files/{hash\_id}/tags  # Добавить теги: {"tags": \["person=anna", "art"]}

GET  /api/v1/search?tag=person:anna\&tag=art  # Поиск по тегам

WS   /api/v1/events          # WebSocket: уведомления о новых файлах/тегах

```



> ✅ Все эндпоинты возвращают `application/json` с валидацией через Pydantic \[\[14]].

\## 🗂️ Хранение файлов на диске



```

db/

├── client.master.db        # Основная метадата (таблицы выше)

├── client.mappings.db      # Опционально: выделенная БД для file\_tag\_mappings (масштабируемость)

└── client\_files/           # Физические файлы

&#x20;   ├── f00/                # Подкаталоги по первым 2 символам хэша

&#x20;   │   ├── f00a1b2c3...    # Файл без расширения, имя = полный hex-хэш

&#x20;   │   └── ...

&#x20;   ├── f01/

&#x20;   └── ...

&#x20;   └── fff/

```



\- Файлы переименовываются в `sha256\_hex` при импорте (как в Hydrus) \[\[12]]

\- Расширение не хранится в имени — определяется по `mime\_type` в БД

\- Поддерживается гранулярность: `f00/0/`, `f00/1/`, ... `f00/f/` для >100k файлов \[\[9]]

\## ⚠️ Критические замечания и рекомендации



1\. \*\*SQLite WAL-режим обязателен\*\* для конкурентного доступа:

&#x20;  ```python

&#x20;  # database/connection.py

&#x20;  conn.execute("PRAGMA journal\_mode=WAL")

&#x20;  conn.execute("PRAGMA synchronous=NORMAL")  # баланс скорость/безопасность

&#x20;  conn.execute("PRAGMA cache\_size=-64000")   # 64MB кэш

&#x20;  ```



2\. \*\*Транзакции для пакетных операций\*\*: Добавление 100 тегов к файлу — одна транзакция, не 100.



3\. \*\*Асинхронность везде\*\*: `aiosqlite`, `aiofiles`, `async def` в сервере — чтобы не блокировать при I/O.



4\. \*\*Валидация на границах\*\*: Pydantic модели для всех входных/выходных данных — защита от некорректных запросов.



5\. \*\*Логирование и мониторинг\*\*: `structlog` + метрики (количество файлов, время поиска) — для отладки и оптимизации.



6\. \*\*Тестирование\*\*: 

&#x20;  - Юнит-тесты на хэширование и валидацию

&#x20;  - Интеграционные: клиент → сервер → БД → проверка результата

&#x20;  - Нагрузочные: 10k файлов, поиск по 5 тегам
